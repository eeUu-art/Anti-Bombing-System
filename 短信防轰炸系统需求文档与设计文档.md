# 短信防轰炸系统需求文档与设计文档

## 一、需求文档

### 1. 项目背景

随着业务发展，短信验证码作为身份校验核心手段，面临恶意 “短信轰炸” 风险 —— 攻击者通过高频请求、多次失败校验等方式，消耗短信资源、干扰业务正常运行，甚至导致用户体验下降与成本浪费。为解决此问题，需基于现有数据库表结构，构建一套覆盖 “场景 - 规则 - 记录” 全链路的短信防轰炸系统，实现精细化、可配置的短信发送与校验管控。

### 2. 需求概述

基于 `sys_sms_scene_strategy`（短信场景策略）、`sys_sms_strategy_rule`（短信规则明细）等表，结合图形验证码联动逻辑，实现以下核心目标：

1. 按业务场景（如登录、注册）配置差异化防轰炸策略；
2. 支持频率（分钟 / 小时 / 天）、失败次数双维度规则管控；
3. 实现区域区号限制、成功校验后计数清零等辅助控制；
4. 留存校验记录用于追溯，同时配置自动清理策略；
5. 联动图形验证码前置校验，提升防轰炸安全性。

### 2. 详细设计

#### 2.1 场景规则服务设计

* **核心功能**：场景与规则的 CRUD、规则关联关系维护
* **关键逻辑**：

1. 场景创建时，校验 strategy\_types 格式（需为 3 位编码逗号分隔，且对应 sys\_sms\_strategy\_rule 存在）；
2. 规则编辑时，若规则已关联场景，需同步检查场景的 strategy\_types 有效性；
3. 提供 “规则生效状态” 开关，禁用规则不参与校验流程。

#### 2.2 校验服务设计

* **核心功能**：图形验证码校验、短信频率 / 失败次数校验
* **关键逻辑**：

1. 频率校验公式：

   `当前时间 - time_value（如5分钟） ≤ 记录的time_minute ≤ 当前时间`，统计该范围内 sms\_req\_count 是否＞max\_count；
2. 失败次数校验公式：

   同频率校验时间范围，统计 error\_count 是否＞fail\_threshold，超限则生成 “锁定标识”（Redis 缓存，过期时间 = time\_value）；
3. 图形验证码校验：

   匹配 scene\_code+identifier+encrypt\_verify\_code，检查 expire\_time 是否未过期、error\_count 是否未超限。

#### 2.3 记录服务设计

* **核心功能**：记录生成、计数更新、查询追溯
* **关键逻辑**：

1. Redis 版本计数方案：

* 频率计数 Key：`sms:freq:{scene_id}:{identifier}:{strategy_id}`，过期时间 = time\_value（如 5 分钟），通过 `INCR`实现计数；
* 失败次数 Key：`sms:fail:{scene_id}:{identifier}:{strategy_id}`，过期时间 = time\_value，通过 `INCR`实现计数；

1. 数据库版本计数方案：

* 按 “identifier+scene\_id+strategy\_id+time\_minute” 唯一键更新记录，避免重复创建；
* window\_end\_time = 创建时间 + 25 小时（覆盖自然日统计场景）。

#### 2.4 清理服务设计

* **核心功能**：历史记录自动清理
* **实现方案**：

1. 定时任务：使用 XXL-Job 调度，每天凌晨 2 点执行清理脚本；
2. 清理 SQL（数据库版本）：

* 清理 sys\_verify\_code\_record：`DELETE FROM sys_verify_code_record WHERE create_time < DATE_SUB(NOW(), INTERVAL 3 DAY);`
* 清理 sys\_sms\_strategy\_rule\_detail：`DELETE FROM sys_sms_strategy_rule_detail WHERE window_end_time < DATE_SUB(NOW(), INTERVAL 3 DAY);`

1. Redis 版本：无需清理，依赖 Key 过期自动删除。
